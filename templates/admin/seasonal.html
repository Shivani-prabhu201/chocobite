🤖 How the ML Seasonal Offers Work
User Side:
1. User opens Customize → Seasonal Offers tab on the store.
2. Frontend calls /api/seasonal-offers — checks the in-memory cache first.
3. If cache is empty (first visit or after refresh) → ML runs on live DB data.
4. User sees 7 offers, clicks Redeem → buy wizard → order placed with is_seasonal_offer:true.
5. A floating notification bubble shows 1 random offer on every page.

ML Algorithm (sklearn KMeans):
1. Features per product: [purchases_this_season, total_purchases, rating, price]
2. StandardScaler normalises all features so price doesn't dominate.
3. KMeans(k=3) clusters products into 3 groups based on purchase behaviour.
4. Clusters are sorted by seasonal purchase count → mapped to discount tiers:
   • Cluster 0 (Popular): 14–17% — already sells well, small nudge.
   • Cluster 1 (Medium): 18–20% — moderate push.
   • Cluster 2 (Low sellers): 21–24% — needs boost, highest discount.
5. Up to 2–3 products per category for diversity.
6. Self-learning: Every new order changes the purchase counts → next refresh re-clusters automatically.

Cache: Offers are cached per season in server memory. They persist until you click Refresh ML, the season changes, or the server restarts. Admin pins/removes override the ML result.
